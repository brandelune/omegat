Glossaries are terminology files. You can create and update them manually, or import them from other projects.

In an OmegaT project, only the default glossary is writable from the application. However the project can also contain any number of other reference glossaries.

All terms in a segment with a match in any of the glossaries attached to the project will be displayed in the Glossary pane.

Glossaries folder

By default, each project contains a glossary subfolder to store the writable glossary and any reference glossaries you want to add to the project.

You can change the name and location of this folder from the project properties dialog. If you have several reference glossaries you use in all your projects, for example, you could create a /Documents/Terminology/ folder and assign it as the glossaries folder for all your projects. You can also use subfolders to organize your glossaries by domain (/Documents/Terminology/ <domain> ) or client (/Documents/Terminology/ <client> ) and use the subfolder appropriate to your project as the glossaries folder.

For the sake of simplicity, explanations in this manual will use the default glossary/ subfolder in the project folder to refer to the glossaries folder assigned to the project.

Default glossary

The default writable glossary is located in the glossary/ subfolder of the project and called glossary.txt.

You can change its name and location in the project properties dialog, but you must give it a .txt or .utf8 extension, and store it in the glossary/ folder or in one of its subfolders.

Continuing the above example, you could set a domain-specific-glossary.txt or client-specific-glossary.txt file as the writable glossary in the applicable domain or client folder, or in a subfolder thereof. This gives you significant flexibility in terms of organizing your glossary files in a way that best suits your workflow.

You do not need to prepare the writable glossary file in advance. If it does not exist when you set up your project, it will be created the first time you add an entry to the glossary. If you choose to use an existing file as the default glossary, check its format beforehand. OmegaT will record all new entries in tab-separated format, and save the file in UTF-8 encoding without checking the original format or character set.

Usage

To use an existing glossary, simply place it in the glossary folder after creating the project. OmegaT automatically detects glossary files in this folder when you open a project.

To add a new term to the writable glossary use Edit > Create Glossary Entry ( Ctrl+Shift+G ). New terms are recognized as soon as they are added. If you want to add terms to other reference glossaries, you will have to edit them in an external text editor. Again, the changes will be recognized as soon as the modified file is saved.

Source terms can be multi-word expressions.

By default, the glossary function uses stemming to find matches. You can deactivate the Use stemming for glossary entries option in the preferences if you want to find only exact matches for terms. You can also adjust various glossary behavior and display options in the Glossary section of the Preferences dialog.

In the Glossary pane, The source term is displayed before the " = " sign and the target terms after it. If the entry contains comments, they are numbered and displayed on separate lines below the terms. Terms found in the writable glossary are shown in bold, while those from other glossaries are presented as regular text.

Use the appropriate shorcut key ( Escape for macOS, Ctrl+Space for other platforms) to display the Autocompletion context menu for glossary matches.

You can use the Mark Glossary Matches option in the View menu to underline source terms in the current segment that have a match in one of the glossaries. You can then right-click on the underlined term and select the desired target term to insert it at in your translation at the current cursor location.

File format

OmegaT glossary files are simple plain text files containing three-column lists, with the source term in the first column, an optional target term in the second column, and an optional comment in the third column.

Plain text glossaries can be "tab-separated values" (TSV) or "comma-separated values" (CSV) files. Reference glossaries can also use the TermBase eXchange *(TBX) ISO standard format.

The writable glossary for the project is always a TSV file saved in UTF-8 encoding.

The encoding used to read reference glossaries depends on their file extension:

Table C.1. Format, extensions and expected encoding
Format Extension Encoding
TSV .txt UTF-8
TSV .utf8 UTF-8
TSV .tab OS default encoding
TSV .tsv OS default encoding
CSV .csv UTF-8
TBX .tbx UTF-8

All glossaries must be located in the glossary subfolder. Glossaries located in nested folders are also recognized.

Common issues

Problem: No glossary terms are displayed.

Probable causes:

  • There is no valid glossary file in the glossary folder.

  • The glossary file is empty.

  • Entries in a plain text glossary file are not separated by a TAB character.

  • The glossary file does not have a recognized extension.

  • The stemming option is disabled, and there is no exact match between the source text in the document and the glossary entry.

  • The glossary file is not in the correct encoding.

  • The current segment does not contain any terms with a match in any of the glossaries.

  • The project has not been reloaded after fixing the probable cause.

Problem: Some characters fail to display correctly in the glossary pane, but display properly in the Editor pane.

  • The extension and file encoding do not match.
